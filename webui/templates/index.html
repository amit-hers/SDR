Usage Examples

      

        
Send a file over RF:

        cat data.bin | ./link_tx --freq 434.0 --mod QPSK
        ./link_rx --freq 434.0 --mod QPSK > received.bin

        
Measure throughput:

        dd if=/dev/urandom bs=1k | ./link_tx --freq 434.0
        ./link_rx --freq 434.0 | pv > /dev/null

        
UDP tunnel (wireless bridge):

        ./link_tx --source udp --port 5005
        ./link_rx --sink udp --sink-ip 10.0.0.1 --port 5006
        nc -u 127.0.0.1 5005 < payload.bin

        
Full duplex (separate TX/RX freqs):

        Device A: --freq 434.0 (TX)  --freq 439.0 (RX)
        Device B: --freq 439.0 (TX)  --freq 434.0 (RX)
      

    

  







  

    

      
AES-256-CTR Frame Encryption

      

        Each frame payload is encrypted with AES-256-CTR before transmission.
        The frame counter is used as the IV, preventing replay attacks.
        Both TX and RX must use the same key.
      


      

        
        

          
          
        

      


      

        

          
          
          
Must match on TX and RX. Never use the all-zero default in production.

        

        

          
        

      


      

        
      

    


    

      
Security Notes

      

        ⚠ Key management: Share the key securely before deployment.
        Never transmit the key over RF.


        ✓ AES-256-CTR provides strong confidentiality for payload data.
        Frame headers (sync word, length) are unencrypted for synchronization.


        CRC-32 is calculated over the encrypted payload,
        detecting transmission errors before decryption.


        Frame counter as IV prevents replay attacks within a session.
        Counter resets on restart — for long-term security, rotate keys periodically.
